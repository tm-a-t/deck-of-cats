# Telegram Bot UX Plan

Дата: 2026-03-01  
Проект: `deck-of-cats/bot`

## 1) Цель

Сделать Telegram-бота удобным для ежедневной работы над задачами разработки:

- Быстро создавать задачу.
- Понимать, что происходит с задачей прямо сейчас.
- Принимать решение по PR без копания в длинных UUID и техническом шуме.

## 2) Ключевое предположение

Одновременно открытых задач мало (обычно `0-10`).  
Это позволяет строить UX вокруг коротких списков и карточек задачи, а не вокруг сложного поиска и фильтров.

Fallback при росте нагрузки:

- Если открытых задач >10, включаем фильтры по статусу и страницу размером 10.
- Добавляем поиск по `public_id`.
- В групповом чате дополнительно фильтр “мои задачи”.

## 3) Что учитываем из документации

- `InlineKeyboardButton.callback_data` ограничен `1-64 bytes` -> payload кнопок должен быть коротким, UUID туда не пихаем.
- После нажатия callback-кнопки клиент показывает прогресс, пока бот не вызовет `answerCallbackQuery` -> нужно всегда быстро отвечать на callback.
- `setMyCommands` позволяет задать понятное меню команд -> снижает порог входа.
- `setChatMenuButton` позволяет вынести основной вход в сценарий в нижнюю кнопку меню.
- `editMessageText` позволяет обновлять одну карточку задачи вместо засорения чата множеством сообщений.
- Лимит текста сообщений `1-4096` символов -> статус и логи должны быть короткими и структурными.

## 4) UX-принципы

- Один экран = одно действие.
- Не показывать внутренние идентификаторы и технические детали без запроса.
- Любая задача имеет постоянную “карточку” в чате, которая обновляется.
- По умолчанию показываем “что делать дальше”, а не “что сломалось”.
- Ошибки даем в двух слоях: коротко для всех + “показать детали” для тех, кому надо.

## 5) Информационная архитектура

Главные входы:

- `➕ Новая задача`
- `📂 Открытые`
- `🕘 Последние`
- `⚙️ Настройки`
- `❓ Помощь`

Команды (минимум):

- `/new` — создать задачу (быстрый режим).
- `/tasks` — список открытых задач карточками.
- `/task <public_id>` — открыть карточку.
- `/help` — сценарии и формат запросов.

## 6) Система идентификаторов (вместо UUID в UI)

Текущее состояние: пользователю показывается полный UUID (`abcfb694-15de-...`), это плохо читается.

Новый подход:

- Внутри системы остается UUID.
- Снаружи показываем короткий `public_id`, например `T-042` или `C7K2M`.
- Во всех кнопках и сообщениях используем только `public_id`.
- Формат callback_data: `t:<public_id>:<action>` (коротко, безопасно для лимита).
- Команды принимают `public_id`, UUID остается только в debug/логах.

## 7) Каноничная state machine и права

### 7.1 Status -> Actions -> Next Status

| Internal status | UI label | Доступные действия в карточке | Следующие статусы |
|---|---|---|---|
| `NEW` | `NEW` | `▶️ Запустить`, `🗑 Закрыть` | `CODEX_IMPLEMENT_RUNNING`, `CLOSED` |
| `CODEX_IMPLEMENT_RUNNING` | `RUNNING` | `🔄 Обновить`, `🧾 Логи` | `CODEX_VALIDATE_RUNNING`, `RETRY_SCHEDULED`, `DEAD_LETTER` |
| `CODEX_VALIDATE_RUNNING` | `RUNNING` | `🔄 Обновить`, `🧾 Логи` | `PR_CREATING`, `RETRY_SCHEDULED`, `DEAD_LETTER` |
| `PR_CREATING` | `RUNNING` | `🔄 Обновить`, `🧾 Логи` | `AWAITING_PREVIEW`, `RETRY_SCHEDULED`, `DEAD_LETTER` |
| `AWAITING_PREVIEW` | `WAITING` | `🔄 Обновить`, `🧾 Логи` | `AWAITING_DECISION`, `RETRY_SCHEDULED`, `DEAD_LETTER` |
| `AWAITING_DECISION` | `WAITING_DECISION` | `✅ Принять merge`, `❌ Принять close`, `🧾 Логи` | `DECISION_APPLYING`, `CLOSED` |
| `DECISION_APPLYING` | `RUNNING` | `🔄 Обновить` | `MERGED`, `CLOSED`, `AWAITING_DECISION` |
| `RETRY_SCHEDULED` | `RETRY` | `▶️ Запустить`, `🧾 Логи`, `🗑 Закрыть` | `CODEX_IMPLEMENT_RUNNING`, `DEAD_LETTER`, `CLOSED` |
| `DEAD_LETTER` | `FAILED_FINAL` | `🧾 Логи` | - |
| `MERGED` | `DONE` | `🔗 Открыть PR` | - |
| `CLOSED` | `CLOSED` | `🔗 Открыть PR` | - |

### 7.2 Правила для Merge/Close

- Решения `Merge/Close` доступны только для авторизованных `TELEGRAM_ALLOWED_USER_IDS`.
- Для `AWAITING_DECISION` сначала показываем экран подтверждения (2-step confirm), затем применяем решение.
- Если пользователь без прав нажал кнопку, бот отвечает callback-alert и не меняет состояние.

### 7.3 Edge-case policy

- Если `editMessageText` не сработал (сообщение удалено/устарело), бот отправляет новую карточку и помечает старую как архивную.
- Для устаревших callback (invalid token/signature) бот всегда показывает понятный alert.
- Для двойных кликов используется идемпотентность по `task_id + action`.

## 8) User Stories

### US-01: Быстро создать задачу

Как пользователь, я хочу отправить идею в 1 сообщении и сразу получить карточку задачи, чтобы не помнить формат команд.

Критерии приемки:

- `/new` без аргументов запускает мини-мастер (FSM) из 2-3 шагов.
- `/new <текст>` работает как быстрый режим.
- После создания приходит карточка с `public_id` и действиями.

### US-02: Видеть только актуальные задачи

Как пользователь, я хочу открыть “Открытые задачи” и увидеть короткий список, чтобы быстро выбрать нужную.

Критерии приемки:

- Список показывает `public_id`, статус, заголовок, “обновлено X мин назад”.
- Есть пагинация кнопками (если >5 задач).
- Нажатие на задачу открывает карточку.

### US-03: Понимать текущий прогресс

Как пользователь, я хочу видеть текущий шаг и последний результат, чтобы понимать, жив бот или нет.

Критерии приемки:

- На карточке есть блок `Сейчас: <шаг>`.
- Есть блок `Последнее событие: <summary>`.
- Есть кнопка `Обновить`.

### US-04: Управлять задачей без команд

Как пользователь, я хочу управлять задачей кнопками, чтобы не вводить команды вручную.

Критерии приемки:

- Доступны контекстные действия v1: `▶️ Запустить`, `🔁 Повторить`, `🗑 Закрыть`.
- Для статуса `AWAITING_DECISION`: `✅ Merge`, `❌ Close` через шаг подтверждения.
- После callback пользователь получает мгновенное подтверждение (`answerCallbackQuery`).

### US-05: Быстро принять решение по PR

Как пользователь, я хочу на карточке видеть PR и preview-ссылку, чтобы решить merge/close без лишних переходов.

Критерии приемки:

- На карточке блоки `PR` и `Preview` отображаются условно:
  - `PR: еще не создан` до шага PR.
  - `Preview: еще не готов` до шага preview.
- Preview нормализован до корня домена (`https://deploy-preview-...netlify.app`).
- Кнопки решения есть прямо под карточкой.

### US-06: Получать понятные ошибки

Как пользователь, я хочу короткое объяснение и конкретное следующее действие, чтобы не застревать.

Критерии приемки:

- Ошибка в карточке в формате: `Что случилось` + `Что делать дальше`.
- Кнопка `Показать детали` раскрывает технический текст.

## 9) Целевые экраны (сообщения)

### 9.1 Карточка задачи

Пример:

```text
🧩 Задача T-042 · AWAITING_PREVIEW
Добавить страницу баланса

Сейчас: PREVIEW
Последнее: PR создан
PR: https://github.com/.../pull/123
Preview: https://deploy-preview-3--pirate-islands.netlify.app
Обновлено: 2 мин назад
```

Кнопки:

- `🔄 Обновить`
- `🧾 Логи`
- `✅ Merge` / `❌ Close` (если нужно решение)

### 9.2 Список открытых

Каждая строка:

- `T-042 · PREVIEW · Добавить страницу баланса`
- `T-041 · RETRY · Починить авторизацию`

Кнопки:

- По одной кнопке на задачу.
- `⬅️` / `➡️` для страниц.

## 10) Контент-дизайн (тексты)

- Короткие строки, без полотен.
- Не больше 1-2 смыслов в одном сообщении.
- Канонические UI-лейблы статусов: `NEW`, `RUNNING`, `WAITING`, `WAITING_DECISION`, `RETRY`, `FAILED_FINAL`, `DONE`, `CLOSED`.
- Эмодзи только как визуальные якоря (`🧩`, `✅`, `❌`, `⚠️`, `🔄`), без перегруза.

## 11) План внедрения (без крупных рисков)

### Этап 1 (быстрый выигрыш)

- Ввести `public_id` и отображать его везде в UI.
- Добавить `/tasks` и карточки задач через inline keyboard.
- Убрать UUID из пользовательских сообщений.
- DoD: есть миграция `public_id`, нет UUID в пользовательских ответах, 5 ручных сценариев пройдены.

### Этап 2

- Ввести FSM-мастер создания задачи.
- Сделать “основной режим: одна задача = одно обновляемое сообщение” через `editMessageText`.
- Добавить fallback на новый message при невозможности редактирования.
- Добавить кнопки `Обновить` и `Логи`.
- DoD: edge-cases `stale callback`, `message deleted`, `double tap` обработаны и покрыты тест-кейсами.

### Этап 3

- Добавить персональные настройки отображения (тихие/подробные уведомления).
- Добавить быстрые шаблоны промптов для `/new`.
- Добавить фильтры/поиск для режима 10+ задач.
- DoD: подтверждена деградация UX отсутствует при 20+ открытых задачах.

## 12) Изменения в кодовой базе (концептуально)

- `domain/entities` + `persistence`: добавить `public_id`.
- `task_repository_impl`: методы поиска по `public_id`.
- `telegram/handlers`: новый `/tasks`, `/task`, callback-роуты для карточек.
- `telegram/keyboards`: генератор клавиатур для списка/карточки.
- `notifier`: обновление существующего сообщения карточки.
- `migrations`: новый столбец `public_id` + уникальный индекс.

Отдельный технический документ можно выделить в `TECH_SPEC.md`, чтобы UX-план оставался продуктовым.

## 13) Метрики UX-качества

- `TTFU` (time to first understanding): время от `task_created` до `first_card_seen`.
  - Цель: `p75 < 10s`, `p95 < 20s`.
- `button_usage_rate`: доля действий через кнопки относительно команд.
  - Цель: `> 70%`.
- `dead_end_rate`: доля карточек без клика по CTA в течение 3 минут после обновления.
  - Цель: `-30%` к baseline за 4 недели.
- `messages_per_task`: среднее число новых сообщений на задачу.
  - Цель: снижение за счет `editMessageText` + fallback-политики.

## 14) Риски и ограничения

- Переезд на `public_id` требует миграции старых задач.
- Нужна аккуратная обработка коллизий коротких ID.
- Избыточные push-уведомления могут раздражать -> нужен режим “тихо”.
- Нужна проверка прав в callback-решениях, иначе риск ошибочного merge/close.
